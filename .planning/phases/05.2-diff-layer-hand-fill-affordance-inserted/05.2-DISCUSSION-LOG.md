## Reverse-direction case (recomputed non-empty + on-disk empty)

| Option | Description | Selected |
|--------|-------------|----------|
| INFO log, no drift | Log once per field: "collector resolved chassis.model_name=\"Latitude 7420\" (was \"\" on disk; on-disk file unchanged per LIFE-04 — manually update the YAML if you want to lock this value)". No DiffEntry, no SystemDriftError. User sees the new knowledge and can act. | ✓ |
| Silent | No log, no DiffEntry, no error. Re-run is a complete no-op. Matches 'hand-fill is sacred / collector yields' mental model. Trade-off: user never learns the collector improved. | |
| DiffEntry as informational (new severity) | Emit a DiffEntry with new severity='info'. Requires extending DiffEntry/DiffResult — more invasive; downstream consumers must filter by severity. Probably overkill for this single use-case. | |

**User's choice:** INFO log, no drift (Recommended).
**Notes:** Locked as D-60 in CONTEXT.md. INFO level (not WARNING) — unobtrusive but visible in default CI/dev logs. The log message names the field path, the resolved value, the previous on-disk value (`""`), and a hint that the on-disk file is intentionally unchanged so the user understands LIFE-04 is in effect and they can manually update the YAML if they want to lock the value.

## Signature-position handling (4 callable signatures: networking_sig, sysctl_sig, environment_sig, drives_sig)

| Option | Description | Selected |
|--------|-------------|----------|
| Strict (signatures match-or-don't) | Soft-pair only applies to the 7 scalar dotted-key positions. Signatures still must match exactly. Lower risk of false soft-pair (e.g., two distinct nodes with different sysctl just because one's networking_sig is empty). Matches user-mental-model: users don't hand-edit sysctl_sig. | ✓ |
| Lenient (signatures soft-pairable too) | Empty signature on one side counts as 'can match anything'. Symmetric treatment with scalars. Slightly higher risk of conflating distinct machines if many positions are empty on both sides. | |

**User's choice:** Strict (Recommended).
**Notes:** Locked as D-61 in CONTEXT.md. The conservative choice — users do not hand-edit `sysctl_sig` or its peers, so there is no hand-fill use-case for lenient signature pairing. Lenient treatment would risk silently conflating genuinely distinct machines whose hardware identity differs along the signature axis but happens to be empty on a few scalar positions. The env/sysctl audit todo (`phase-future-env-sysctl-fingerprint-audit.md`) is a separate concern about WHAT goes into the signatures, not about how the diff layer pairs them.

## Pairing algorithm shape

| Option | Description | Selected |
|--------|-------------|----------|
| Two-pass: exact-match first, soft-pair the orphans | (1) Index both sides by exact fingerprint, pair identical fingerprints. (2) Collect the remaining unmatched on-disk + in-memory orphans. (3) Soft-pair: for each in-memory orphan, find unique on-disk orphan whose non-empty fingerprint positions match. (4) Surviving orphans emit DiffEntry per D-46/D-47. Clean separation; easy to test each pass independently. | ✓ |
| Single-pass: try exact, then soft, then orphan in one walk | For each in-memory client, try exact fingerprint match first; if miss, try soft-pair on the remaining on-disk pool; if still miss, emit orphan. Slightly less code but harder to test pairing logic in isolation. Order-sensitive when multiple soft-pair candidates exist. | |

**User's choice:** Two-pass (Recommended).
**Notes:** Locked as D-62 in CONTEXT.md. Two-pass is the cleaner conceptual model and makes each pass independently testable (you can lock the soft-pair pass with unit tests that bypass the exact-match pass entirely). Performance is irrelevant at typical fleet sizes (N×M soft-pair cost where N, M ≤ orphan_count, bounded by fleet size). The single-pass alternative would have made order-sensitivity hidden inside the main loop — harder to reason about when multiple soft-pair candidates exist.
