# Phase 5.2: Code Review Report

**Reviewed:** 2026-06-25
**Depth:** standard
**Files Reviewed:** 2

## Summary

Phase 5.2 inserts the HANDFILL-01 soft-pair affordance into the pure-function diff core. The implementation extracts a new `_emit_leaf_diffs` helper, adds `_soft_pair_orphans` as a pre-pass between exact-match and orphan-emission, and threads a module-level `logger` for D-60 reverse-direction INFO emission. Twelve new tests in `TestDiffHandFillAffordance` lock the documented behavior.

The implementation is generally well-structured, well-commented, and the new tests cover the bulk of the documented contract. **However, the soft-pair pairing algorithm has a greedy-consumption defect that can silently conflate distinct machines under D-63 ambiguity — the exact failure mode D-63 was designed to prevent.** Other findings are quality issues around logging conventions, the documented-vs-actual scope of D-60, and a gap in test coverage for the greedy-conflation edge case.

## Critical Issues

### CR-01: Greedy consumption in `_soft_pair_orphans` can silently conflate distinct machines (violates D-63)

**File:** `mlpstorage_py/system_description/diff.py:299-343`

**Issue:** `_soft_pair_orphans` pairs in-memory orphans against on-disk orphans in a single greedy pass. Once an on-disk fingerprint is consumed by an earlier (sort-order) in-memory orphan as a "unique" candidate, it is removed from the candidate pool for subsequent in-memory orphans via the `if disk_fp in paired_on_disk_fps: continue` skip. This converts genuine D-63 ambiguity into deterministic-but-arbitrary pairings.

Concrete failure scenario (two distinct hand-filled machines, one collector-empty stanza, one partial-empty stanza):

- `on_disk = [{model_name="Dell A", ...same scalars/sigs...}, {model_name="Dell B", ...same scalars/sigs...}]`
- `in_memory = [{model_name="Dell A", cpu_model=""}, {model_name="", cpu_model=""}]`

Walk-through:
1. `mem_fp[0]` (model="Dell A", cpu="") iterates disk_fps. Disk "Dell A" matches (model aligns, cpu empty on mem side, at_least_one_empty=True). Disk "Dell B" does NOT match (model "Dell B" vs "Dell A" both non-empty — scalar_ok=False). Candidates = {DiskA}. Unique. **Pair (Mem0, DiskA), add DiskA to `paired_on_disk_fps`.**
2. `mem_fp[1]` (model="", cpu="") iterates disk_fps. Disk "Dell A" is **skipped because already paired**. Disk "Dell B" matches (all scalars wildcarded on mem side). Candidates = {DiskB}. Unique. **Pair (Mem1, DiskB) — but Mem1 was genuinely ambiguous (it matches both DiskA and DiskB), and per D-63 should have fallen back to orphan emission.**

The greedy order silently chose DiskB for Mem1, conflating two distinct machines. Per the docstring (line 280-283): "D-63 ambiguity rule: if two or more on-disk orphans qualify as candidates for the same in-memory orphan, NEITHER is paired and BOTH sides stay in the remaining-orphan lists. Conservative — never silently conflate distinct machines." The code does not honor this when prior greedy pairings have reduced the candidate pool.

**Fix:** Two-phase algorithm — collect each mem_fp's full candidate set against the *original* on-disk orphan pool (ignoring prior pairings), then commit only those pairings that are globally unique. Concretely:

```python
# Phase A: gather all candidate sets without consuming.
candidate_map: dict[tuple, list[tuple]] = {}
for mem_fp in sorted(in_memory_orphans_by_fp.keys(), key=repr):
    candidates: list[tuple] = []
    for disk_fp in sorted(on_disk_orphans_by_fp.keys(), key=repr):
        if any(mem_fp[i] != disk_fp[i] for i in signature_positions):
            continue
        scalar_ok = True
        at_least_one_empty = False
        for i in scalar_positions:
            if mem_fp[i] == "" or disk_fp[i] == "":
                at_least_one_empty = True
                continue
            if mem_fp[i] != disk_fp[i]:
                scalar_ok = False
                break
        if not scalar_ok or not at_least_one_empty:
            continue
        candidates.append(disk_fp)
    candidate_map[mem_fp] = candidates

# Phase B: pair only when mem_fp has a unique candidate AND that
# candidate is uniquely claimed by this mem_fp (no other mem_fp lists
# it as a candidate).
disk_claim_count: dict[tuple, int] = {}
for cands in candidate_map.values():
    for d in cands:
        disk_claim_count[d] = disk_claim_count.get(d, 0) + 1

for mem_fp, cands in candidate_map.items():
    if len(cands) == 1 and disk_claim_count[cands[0]] == 1:
        paired_fp_pairs.append((mem_fp, cands[0]))
        paired_on_disk_fps.add(cands[0])
        paired_in_memory_fps.add(mem_fp)
```

Add a regression test in `TestDiffHandFillAffordance` mirroring the scenario above — assert `r.empty is False` and that two fingerprint-level orphan entries surface for the unpaired Mem1/DiskB pair.

## Warnings

### WR-01: D-60 INFO log fires on every leaf, not only fingerprint scalar positions

**File:** `mlpstorage_py/system_description/diff.py:388-403`

**Issue:** The docstring (line 86-91, line 254-288) consistently scopes D-60 reverse-direction to "scalar positions" of the fingerprint — the 7 dotted-key scalar fields. The implementation in `_emit_leaf_diffs` applies the rule to **every** leaf in the flattened stanza, including paths the soft-pair contract never considered (e.g. `sysctl[0].name`, `drives[0].vendor_name`, arbitrary new schema keys).

For a soft-paired stanza pair, this can fire on any path where on-disk had `""` and in-memory has a non-empty string — even when that path is not a fingerprint scalar. Worse, on the **exact-match** pass-1 path (common_fps), `_emit_leaf_diffs` is also invoked, so D-60 will fire on common-fingerprint pairs whenever a non-fingerprint scalar leaf changed from `""` to non-empty. That is a behavior change that pre-existed Pass-2 — previously such a path would have been a DiffEntry; now it is silently swallowed with only an INFO log.

This is a real semantic regression for any non-fingerprint leaf that changes from `""` to non-empty: prior to Phase 5.2 it was reported as drift; after Phase 5.2 it is silently logged.

**Fix:** Either (a) scope the D-60 check to fingerprint scalar positions only — pass the set of `(jsonpath → fingerprint-position)` mappings into `_emit_leaf_diffs` and gate the INFO log on path membership; or (b) explicitly document and lock the broadened scope with a test that asserts an arbitrary non-fingerprint `""` → non-empty change produces no DiffEntry. Option (a) preserves prior drift behavior on non-fingerprint paths; option (b) is the smaller code change but the larger semantic shift and should be confirmed against the SER-02 / Pitfall-3 contract owners.

### WR-02: Greedy-conflation gap in test coverage

**File:** `tests/unit/test_diff.py:530-548`

**Issue:** `test_ambiguous_soft_pair_two_candidates_falls_back_to_orphans` covers only the case where *both* in-memory orphans are equally ambiguous (model_name="" for both). It does NOT cover the asymmetric case where one in-memory orphan is uniquely matchable and a *second* in-memory orphan is genuinely ambiguous — the exact configuration that exposes CR-01. The current test passes both pre-fix and post-fix, providing no regression lock against the greedy-pairing bug.

**Fix:** Add a test mirroring CR-01's scenario:

```python
def test_partial_ambiguity_unique_then_ambiguous_does_not_greedily_conflate(self):
    # Mem0 uniquely matches DiskA. Mem1 is ambiguous between DiskA and
    # DiskB. Greedy pairing would consume DiskA on Mem0, then mis-pair
    # Mem1→DiskB. Per D-63, Mem1 must fall back to orphan emission.
    on_disk = [
        _make_node_dict(model_name="Dell A"),
        _make_node_dict(model_name="Dell B"),
    ]
    in_mem = [
        _make_node_dict(model_name="Dell A"),  # exact match — pass 1
        _make_node_dict(model_name=""),         # ambiguous: matches A and B
    ]
    r = diff_node_dict_lists(on_disk, in_mem)
    assert r.empty is False
    assert any("fingerprint=" in e.path for e in r.entries), (
        "D-63: ambiguous orphan must NOT be silently paired to the "
        "leftover candidate after greedy consumption."
    )
```

Note that as written, the first stanza is exact-match (pass-1) so DiskA is not in the orphan pool at all. The greedy bug shows up with two mem_fps where one is *partially* hand-filled enough to be unique and the other is fully blank. Adjust to ensure both mem_fps are orphans — e.g. make Mem0 have cpu_model="" so it falls out of exact match but still uniquely pairs with DiskA via cpu_model match elsewhere. The reviewer's CR-01 scenario is the load-bearing template.

### WR-03: Pure-function module claim broken by mutating helper signature

**File:** `mlpstorage_py/system_description/diff.py:361-365`

**Issue:** The module docstring (line 7-9) emphasizes "pure transformation: list of dicts in, structured diff out." `_emit_leaf_diffs` accepts a caller-owned `entries: list[DiffEntry]` and mutates it via `.append()`. This is a private helper but breaks the stated discipline and complicates reasoning when reading call sites in `diff_node_dict_lists` (you must scroll to the helper to discover that `entries` grows). Pure functional style would return a new list:

**Fix:**
```python
def _emit_leaf_diffs(
    on_disk_stanza: dict,
    in_memory_stanza: dict,
) -> list[DiffEntry]:
    entries: list[DiffEntry] = []
    ...
    return entries

# Callers:
for fp in sorted(common_fps, key=repr):
    entries.extend(_emit_leaf_diffs(on_disk_by_fp[fp], in_memory_by_fp[fp]))
```

Trivial mechanical change; restores the pure-function discipline. Lower-priority than CR-01/WR-01/WR-02 but worth doing while the code is open.

### WR-04: `logger.info` uses f-string instead of deferred `%`-style formatting

**File:** `mlpstorage_py/system_description/diff.py:398-402`

**Issue:** The D-60 log call always evaluates the f-string even when the `mlpstorage_py.system_description.diff` logger is filtered below INFO. The standard `logging` convention is `logger.info("template %s=%r ...", path, mem_v)` so formatting is deferred to the handler. With f-strings, every soft-pair-eligible diff pays the formatting cost regardless of effective log level — and the `mem_v!r` call on long sysctl tuples is non-trivial.

**Fix:**
```python
logger.info(
    "collector resolved %s=%r (was \"\" on disk; on-disk file "
    "unchanged per LIFE-04 — manually update the YAML if you want "
    "to lock this value)",
    path, mem_v,
)
```

This also matches the project convention referenced in the comment at line 113 (`logging.getLogger(__name__)` style).

## Info

### IN-01: `_soft_pair_orphans` does not directly assert it is operating on disjoint dicts

**File:** `mlpstorage_py/system_description/diff.py:258-261`

**Issue:** The function assumes its input dicts contain only orphans (no shared fingerprints with the common set), but does not assert or validate this. A future caller misuse — passing the full `on_disk_by_fp` / `in_memory_by_fp` dicts — would silently re-pair stanzas already in the common-fp set, producing duplicate DiffEntries. A one-line `assert` or a docstring callout strengthens the contract.

**Fix:** Add a docstring line stating "Inputs MUST be orphan-only (no fingerprints present on both sides)." Optionally `assert not (set(in_memory_orphans_by_fp) & set(on_disk_orphans_by_fp))` for defense-in-depth.

### IN-02: `key=lambda p: repr(p[0])` recomputes large `repr` strings per comparison

**File:** `mlpstorage_py/system_description/diff.py:468`

**Issue:** `sorted(paired_fp_pairs, key=lambda p: repr(p[0]))` builds `repr()` of each in-memory fingerprint tuple for sorting. For large fleets with long sysctl tuples, the repr can be hundreds of characters. Not a correctness issue (performance is explicitly out of v1 scope), but the key function would be cleaner as `key=lambda p: repr(p[0])` only being computed once per item if Python's sort algorithm caches keys — which it does (Timsort key extraction). So this is fine; flagging only because the repeated computation pattern reads as suspicious. No action needed unless profiled.

### IN-03: D-60 docstring "scalar position" wording is ambiguous

**File:** `mlpstorage_py/system_description/diff.py:86-91`

**Issue:** The Phase 5.2 docstring extension (lines 86-91) says D-60 fires "when on-disk is '' at a scalar position and the collector finally resolves a non-empty value". Reading this in isolation suggests scope is the 7 fingerprint scalar positions; the code applies it to every flattened leaf (see WR-01). Whichever direction WR-01 lands on (scope D-60 to fingerprint scalars, or formally broaden it), the docstring should be updated to match — currently it reads as scoping while the implementation does not scope.

**Fix:** Update lines 86-91 to read "...at any leaf path where on-disk holds the empty string and the in-memory value is a non-empty string..." OR (preferred) tighten the implementation per WR-01.
