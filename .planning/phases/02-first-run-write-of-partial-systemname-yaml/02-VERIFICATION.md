LIFE-01 — the lifecycle wiring — is **partially blocked** by CR-01, a real defect in the Phase 2-05 hook integration at `mlpstorage_py/benchmarks/base.py:991`. The hook reads `self._cluster_info_start` directly, but `_collect_cluster_start()` has an early-return branch (lines 634-636) that fires for two production paths:

1. **Any `datagen`/`configview` command** — the gating predicates `_should_collect_cluster_info()` (line 477) and `_should_use_ssh_collection()` (line 567) both short-circuit to False on those commands, leaving `_cluster_info_start` uninitialized. `Benchmark.__init__` does not initialize the attribute either. Evaluating the argument at line 991 raises AttributeError, which the catch-all `except Exception` at line 997 relabels as "Failed to write systemname.yaml" and re-raises — aborting datagen before any work happens. **This is a regression from Phase 1**: the datagen path used to complete normally, and the phase goal explicitly contracts that datagen must NOT write the file. The current implementation violates that contract by crashing instead of cleanly skipping.

2. **VectorDB `run` without `--hosts`** — `cli/vectordb_args.py:107` sets `default=None` for `--hosts`. Same gating, same early-return, same AttributeError. The VectorDB user story is a real production path (the benchmark is listed in CLAUDE.md, and `TestVectorDBSystemnameYamlHook::test_vectordb_run_writes_systemname_yaml` exists to assert it). The unit test masks the bug because it mocks `_collect_cluster_start` with a side-effect that explicitly seeds `_cluster_info_start`.

All 12 integration tests pass, all 73 kvcache+vectordb unit tests pass, and the 257-test Phase 02 verification suite is green — but every test in the suite that exercises `bench.run()` does so through the `_make_benchmark` helper (or the equivalent fixture in the per-benchmark hook test) which always mocks `_collect_cluster_start` to install the attribute. The bug is invisible to the entire test suite as written.

Two existing tests in `tests/unit/test_benchmarks_base.py` (`test_validate_environment_called_on_run` at line 619 and `test_run_calls_start_and_end_collection` at line 938) DO exercise the unmocked path and would have caught CR-01 — but the test module currently fails at collection time due to a missing `psutil` import (deferred per the SUMMARY's "deferred items #1"). Fixing that import OR adding a regression test that omits the `_collect_cluster_start` mock would lock the fix.
